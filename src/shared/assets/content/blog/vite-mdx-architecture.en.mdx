# Managing content in Markdown and serving it with Vite

When you start building a portfolio or a blog and don't want to deal with server infrastructure, you often face a dilemma:

- Use a **CMS via SaaS**.
- Use a framework like **Next/Nuxt** which offers dozens of content editing tools.
- Turn to recent **Git-based CMS** solutions.

In my case, although the heart of the site is based on Markdown content, I wanted to keep it simple by limiting external tools since I won't have hundreds of articles to manage. With this in mind, I discovered a simple solution based on **Vite** and the **MDX** format.

<span className="text-sm text-muted-foreground block border-l-2 border-primary/20 pl-4 italic mb-6">
  MDX (Markdown Executable) is an extension of traditional Markdown that allows
  you to insert React components (JSX) directly into your text files. Vite, for
  its part, is the successor to Webpack: it provides an instantaneous
  development server and a high-performance bundler.
</span>

The idea is simple: each project or article is an `.mdx` file stored directly in the codebase. No database, no external API. This allows me to write my content using simple syntax while having the ability to display web components when needed. This content is then transformed into JS modules by Vite.

## The secret: `import.meta.glob`

The heart of the system relies on a powerful Vite feature: `import.meta.glob`. It allows you to scan a folder and automatically generate an index of dynamic imports.

```typescript
const modules = import.meta.glob("./content/*.mdx");
```

Unlike a classic import that would load all content at startup, `import.meta.glob` is **lazy**. It returns an object like this:

```typescript
{
  "./content/project-1.mdx": () => import("./content/project-1.mdx"),
  "./content/project-2.mdx": () => import("./content/project-2.mdx")
}
```

In concrete terms, Vite crawls my application code and creates two things:

1.  **An import function list** for modules that indexes all `.mdx` files found in the specified folder.
2.  **On-demand loading**: each function allows loading a module containing the article's content as a React component along with its metadata.

## From indexing to rendering

To display the list of articles on the homepage, the application first needs to know what’s "in stock." The system is divided into two steps:

1.  **Indexing**: We loop through all the import functions listed by Vite to retrieve their metadata. Technically, the browser downloads each article's compiled file. It's very fast because these are just small fragments of JS code, but it means the data is already "there" behind the scenes.
2.  **Deferred Rendering (Lazy Loading)**: Even if the code is downloaded, it is never executed unnecessarily. Thanks to React's `lazy()`, the article component is only instantiated and displayed if the user actually clicks on it (thus avoiding premature loading of images or heavy resources).

## But how do we retrieve the metadata?

During the build process, two **Remark** plugins (`remark-frontmatter` and `remark-mdx-frontmatter`) intercept the file.

They transform the YAML block into a simple JavaScript variable named `frontmatter`, which is automatically added to the file's exports. When my code performs an `import`, it doesn't receive raw text, but a ready-to-use JavaScript object. It’s fast, typed, and perfectly integrated into the application's lifecycle.

## Why this system?

- **Pragmatism and Performance**: For a portfolio, the download overhead is negligible compared to the gain in simplicity. If the site were to become massive, I could simply generate a JSON index at build time. For now, Vite's efficiency makes this optimization unnecessary.
- **Developer Experience**: Writing an article is just about creating a Markdown file. I benefit from the power of React (displaying web components if necessary) and I have full control to handle display automatically based on the selected language.
- **Zero Maintenance**: No need for a database server or managing CMS access. Everything is versioned in Git.

Nothing revolutionary overall, but it allows me to manage my content exactly how I want, without having to learn complex tools or hack around limitations like MDX support or internationalization.
