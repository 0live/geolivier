# Managing Content with Vite and MDX: Simplicity for Performance

When you start building a portfolio or a blog, you often find yourself facing a dilemma: use a heavy CMS (like WordPress or Contentful) or code every page by hand. On this site, I opted for a third way, both simple and powerful: a system based on **Vite** and **MDX**.

The idea is simple: each project or article is a `.mdx` file stored directly in the code. No database, no external API, just files. But wait, without exploding the initial weight of the application.

## The Secret: `import.meta.glob`

The heart of the system relies on a great Vite feature: `import.meta.glob`. It allows scanning a folder and retrieving a list of dynamic import functions.

```typescript
const modules = import.meta.glob("./content/*.mdx");
```

What's powerful is that Vite doesn't load the content of all files at once. It just gives us the "keys" to load them later.

## Metadata and Lazy Loading

## From Indexing to Rendering

To display the list of articles on the home page, the application first needs to know what's "in stock". This is where the system splits into two steps:

1.  **Indexing (`fetchAll`)**: We loop through all the import functions listed by Vite (the `import.meta.glob`) to retrieve their metadata. Technically, the browser downloads the compiled JS file for each article. It's very fast because these are just small code fragments, but it means the data is already "there" behind the scenes.
2.  **Deferred Rendering (Lazy Loading)**: Even if the code is downloaded, it is never executed unnecessarily. Thanks to React's `lazy()`, the article component is only instantiated and displayed if the user actually clicks on it.

It's an ideal compromise: we fetch the necessary catalog info all at once, but we only make the processor work for display when it's truly needed.

## But how do we get the metadata?

One might think that we need to scan the text file to look for the YAML block. In reality, it's much smarter. During the build, two **Remark** plugins (`remark-frontmatter` and `remark-mdx-frontmatter`) intercept the file.

They transform the YAML block into a simple JavaScript variable named `frontmatter`, which is automatically added to the file's exports. When my code does an `import`, it doesn't receive text, but a ready-to-use JavaScript object. It's fast, typed, and perfectly integrated into the application's lifecycle.

## Why this system?

1. **Pragmatism and Performance**: For a portfolio, the download overhead is negligible compared to the gain in simplicity. If the site were to become massive (thousands of articles), we could simply generate a JSON index during build time to load MDX modules only on click. For now, Vite's lightness makes this optimization unnecessary!
2. **Developer Experience**: Writing an article is just creating a Markdown file. We benefit from the power of React (we can put interactive components, like my MapLibre maps, directly in the text).
3. **Zero Maintenance**: No need for a database server or managing CMS access. Everything is versioned in Git.

This is nothing revolutionary, it's just the intelligent use of modern tools. Sometimes, the simplest solution is also the most elegant!
