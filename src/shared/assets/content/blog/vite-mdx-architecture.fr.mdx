# Nouveau chapitre, nouveau projet

Après avoir travaillé plus de 7 ans pour une ONG, j’ai fait les frais des baisses de <a href="https://www.coordinationsud.org/actualite/etude-impacts-coupes/" target="_blank">financement dans le secteur de l’aide internationale</a>. En tant que développeur, j’intervenais surtout sur des <a href="/projects">projets</a> de suivi/analyse des programmes en lien avec les sièges.

En période de réduction budgétaire, ces projets ont été les premiers à être mis en pause ou supprimés — un choix évidemment compréhensible — ce qui a conduit en décembre 2025 à la fermeture du pôle de développement web auquel j’étais rattaché.

Après quelques semaines de pause, je vais profiter d’avoir un peu de temps libre pour me consacrer à un projet que j’ai en tête depuis un moment : <a href="https://github.com/0live/Canopy" target="_blank">Canopy</a>. Mais avant cela, j’avais envie d’implémenter rapidement un site perso afin d’avoir une carte de visite interactive, mais aussi pour pouvoir disserter librement sur tout ce qui touche au domaine assez particulier de la géomatique.

On commence aujourd’hui avec un sujet purement web. Rien de cartographique ni de révolutionnaire ici, juste une présentation d’une simple feature technique à l’occasion du lancement du site.

# Gérer son contenu en Markdown et le servir avec Vite

Quand on commence à construire un portfolio ou un blog et qu’on n’a pas envie d’y associer une infrastructure serveur, on se retrouve souvent face à un dilemme :

- Utiliser un **CMS via un SaaS**.
- Utiliser un framework type **Next/Nuxt** qui offre des dizaines d’outils d’édition de contenu.
- Se tourner vers les récents <a href="https://www.teotimepacreau.fr/blog/git-based-headless-cms/">**Git-based CMS**</a>.

Dans mon cas, bien que le cœur du site soit basé sur du contenu en Markdown, j’avais envie de rester simple en limitant les outils externes car je ne vais pas avoir des centaines d’articles à gérer. C’est dans cette optique que j’ai découvert une solution simple basée sur **Vite** et le format **MDX**.

<span className="text-sm text-muted-foreground block border-l-2 border-primary/20 pl-4 italic mb-6">
  Le MDX (Markdown Executable) est une extension du Markdown traditionnel qui
  permet d'insérer des composants React (JSX) directement dans vos fichiers
  texte. Vite, de son côté, est le remplaçant de Webpack : il fournit un serveur
  de développement instantané et un bundler performant.
</span>

L'idée est simple : chaque projet ou article est un fichier `.mdx` stocké directement dans le code. Pas de base de données, pas d'API externe. Cela me permet d’écrire mon contenu avec une syntaxe simple tout en ayant la possibilité d’afficher des composants web si besoin. Ce contenu est ensuite transformé en modules JS par Vite.

## Le secret : `import.meta.glob`

Le cœur du système repose sur une fonctionnalité puissante de Vite : `import.meta.glob`. Elle permet de scanner un dossier et de générer automatiquement un index d'importations dynamiques.

```typescript
const modules = import.meta.glob("./content/*.mdx");
```

Contrairement à un import classique qui chargerait tout le contenu au démarrage, `import.meta.glob` est **paresseux (lazy)**. Il renvoie un objet de ce type :

```typescript
{
  "./content/projet-1.mdx": () => import("./content/projet-1.mdx"),
  "./content/projet-2.mdx": () => import("./content/projet-2.mdx")
}
```

Concrètement, Vite parcourt le code de mon application et crée deux choses :

1.  **Une liste de fonctions d'importation** de modules qui indexe tous les fichiers `.mdx` trouvés dans le dossier spécifié.
2.  **Un chargement à la demande** : chaque fonction permet de charger un module contenant le contenu d'un article sous forme de composant React ainsi que ses métadonnées.

## De l'indexation au rendu

Pour afficher la liste des articles sur la page d'accueil, l'application doit d'abord savoir ce qu'il y a "en stock". Le système se divise en deux étapes :

1.  **L'indexation** : On boucle sur toutes les fonctions d'importation listées par Vite pour récupérer leurs métadonnées. Techniquement, le navigateur télécharge le fichier compilé de chaque article. C'est très rapide car ce ne sont que des petits fragments de code JS, mais cela signifie que la donnée est déjà "là", en coulisses.
2.  **Le rendu différé (Lazy Loading)** : Même si le code est téléchargé, il n'est jamais exécuté inutilement. Grâce au `lazy()` de React, le composant de l'article n'est instancié et affiché que si l'utilisateur clique réellement dessus (évitant ainsi le chargement d'images ou de ressources lourdes prématurément).

## Mais comment récupère-t-on les métadonnées ?

Lors du build, deux plugins **Remark** (`remark-frontmatter` et `remark-mdx-frontmatter`) interceptent le fichier.

Ils transforment le bloc YAML en une simple variable JavaScript nommée `frontmatter`, qui est automatiquement ajoutée aux exports du fichier. Quand mon code fait un `import`, il ne reçoit pas du texte brut, mais un objet JavaScript déjà prêt. C'est rapide, typé et parfaitement intégré au cycle de vie de l'application.

## Pourquoi ce système ?

- **Pragmatisme et Performance** : Pour un portfolio, le surpoids au téléchargement est négligeable par rapport au gain de simplicité. Si le site devenait géant, il suffirait de générer un index JSON au build. Pour l'instant, la légèreté de Vite rend cette optimisation superflue.
- **Expérience développeur** : Écrire un article, c'est juste créer un fichier Markdown. On profite de la puissance de React et j’ai la main pour gérer l’affichage automatiquement en fonction de la langue sélectionnée.
- **Zéro maintenance** : Pas besoin de serveur, de base de données ou de gérer des accès CMS. Tout est versionné dans Git.

Rien de révolutionnaire dans l’ensemble, mais cela me permet de gérer mon contenu exactement comme je le veux, sans outils complexes ni bidouillages pour l'internationalisation.
